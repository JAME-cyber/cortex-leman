⚡ Question audit type : « Pourquoi NATS et pas Kafka ? »
Réponse : NATS JetStream est plus léger, plus rapide (latence sub-millisecond), parfait pour l'event-driven pair-à-pair. Kafka serait surdimensionné pour notre scale. NATS offre le streaming persistant (JetStream) avec replay, exactement ce qu'il nous faut pour le journal d'audit.
03 — Les 5 Agents

Chaque agent a un rôle précis

Tous héritent de BaseAgent, souscrivent au bus NATS, et publient leurs résultats

📐 BaseAgent — La classe abstraite commune

Tous les agents héritent de BaseAgent. Le cycle de vie est identique :

1. start() — Connexion au bus

S'abonne aux sujets NATS définis dans subscribe_subjects. Chaque agent a sa propre queue group ("agent-{name}").

2. _handle_message() — Réception protégée

Vérifie le circuit breaker. Si bloqué → ignore. Sinon → appelle process().

3. process() — Traitement métier ABSTRAIT

Chaque agent implémente SA logique. C'est le cœur de l'agent.

4. _publish_result() — Publication

Publie le résultat sur cleman.agent.result avec intention_id, agent_source, timestamp.

# Le pattern clé — chaque agent suit ce modèle :
class BaseAgent(ABC):
    def __init__(self, name, subscribe_subjects):
        self.name = name
        self.agent_id = f"{name}-{uuid4().hex[:8]}"

    async def _handle_message(self, data, meta):
        # 1. Vérifier circuit breaker
        if not cb.allow_request(): return
        try:
            result = await self.process(data, meta)  # ABSTRAIT
            cb.record_success()
            await self._publish_result(result)
        except:
            cb.record_failure()  # → le circuit breaker s'ouvre

📊 Agent Data — Recherche & Collecte

Sujet : cleman.data.query + cleman.data.complement

L'Agent Data est le "chercheur" du système. Son pipeline en 6 phases :

Phase 0: Data Residency

Vérifie si la verticale autorise les recherches externes. En mode Haute Protection: local uniquement. Avocat/Banque/Santé: avertissement si cloud.

Phase 1: Reformulation LLM

Le LLM reformule la requête pour optimiser la recherche. Si LLM indisponible → utilise la requête brute (mode dégradé).

Phase 2: RAG Vectoriel (ChromaDB)

Recherche sémantique dans les embeddings. Retourne les chunks les plus pertinents avec score de distance.

Phase 3: Knowledge Vault (plein texte)

Recherche dans les documents stockés du client. Vault = espace de stockage isolé par client.

Phase 4 & 5: Extraction structurée LLM

Le LLM extrait les faits des sources bruts en JSON structuré. Chaque fait a: description, source, confidence, timestamp, ID unique.

Phase 6: Compilation

Score de confiance = 40% pertinence sources + 60% confiance faits. Identification des risques.

💬 En audit : « L'Agent Data sépare la recherche (faits) de l'analyse (opinions). Il ne produit JAMAIS d'avis, uniquement des faits sourcés et horodatés. Le mode dégradé (sans LLM) retourne les sources brutes. »

🧠 Agent Raisonnement — Analyse & Recommandations

Sujet : cleman.reasoning.analyze

C'est le "cerveau" du système. Pipeline en 5 phases :

Phase 1: Analyse rapide programmatique

Analyse contextuelle SANS LLM. Classification de la requête (comparison, risk_assessment, recommendation, general). Score de confiance de base (0.75).

Phase 2: Analyse LLM

Appelle le LLM avec un skill prompt métier + contexte RAG. Le prompt demande du JSON structuré: textes applicables, confidence, risques, recommandations. Utilise le routing par verticale (Claude pour juridique, GPT-4o pour santé, etc.).

Phase 3: Recommandations fusionnées

Fusion des résultats programmatiques + LLM. Si non conforme → requires_review.

Phase 4: ★ Reflection Node (JP Morgan) ★

Inspiré du pattern "Ask David" de JP Morgan. Un DEUXIÈME appel LLM en mode "réviseur" critique l'analyse. Il peut: confirmer (delta < 0.3) ou réviser à la baisse. Les issues trouvées sont ajoutées aux risques.

v5.3: Injecte les échecs passés (FailureMemory) et les leçons apprises (ProceduralMemory) dans le prompt de critique. Les améliorations détectées alimentent automatiquement la mémoire procédurale.

Phase 5: Triggers de révision

Si incohérence détectée → publie sur cleman.intention.revise pour reformuler la requête.

# Le calcul de confiance finale ( Pondération ):
final = 0.4 × confidence_programmatique + 0.6 × confidence_llm
# Le Reflection Node peut ajuster ce score à la baisse
# s'il détecte des faiblesses dans l'analyse.

⚡ Agent Action — Exécution avec Saga Pattern

Sujet : cleman.action.execute

Le seul agent qui exécute des actions. Protégé par 3 verrous :

🔒 Verrou 1: Gel Médiateur

Si l'intention est gelée (complet OU dégradé) → l'Action est bloqué. Il ne PEUT PAS exécuter. Le code vérifie intention_store.is_action_blocked() AVANT tout traitement.

🔒 Verrou 2: Verrou distribué Redis

Acquiert un lock Redis sur intention:{id}. Empêche 2 agents d'exécuter la même intention simultanément. Auto-expiration 30s.

🔒 Verrou 3: Circuit Breaker

Si l'Action échoue 5 fois → circuit OPEN → bloque tout pendant 60s → HALF_OPEN → teste la récupération.

Sujets : cleman.validate.request + écoute agent.result, mediator.conflict, mediator.freeze, arbitration.decision

Le Superviseur est l'observateur permanent. Il ne prend PAS de décision — il prépare les dossiers et alerte.

📋 Health Board

Maintient un tableau de bord par intention: confiance globale, agents ayant répondu, conflits détectés, gels, inactivité.

💬 En audit : « Le Superviseur est un observateur passif. Il ne bloque JAMAIS rien. Il journalise chaque observation et alerte si dégradation. Il prépare les dossiers d'arbitrage. C'est un watchdog, pas un décideur. »
04 — Le Médiateur

🛡️ Le cœur de la confiance

100% déterministe — 0% LLM — Jamais une IA ne décide seule

💬 En audit, tu dis :
« Le Médiateur est un agent transverse qui intercepte TOUS les résultats des autres agents en temps réel. Il utilise un moteur de règles JsonLogic — c'est du code déterministe, PAS de LLM. Si une règle est violée, il gèle automatiquement l'intention et demande un arbitrage humain obligatoire. Il existe deux modes de gel: complet (tous les agents bloqués) ou dégradé (seul l'Action est bloqué, Data et Raisonnement continuent d'enrichir le dossier pour l'arbitre). »

🔍 Détection de conflits

Le Médiateur compare les positions des agents entre eux. Conflit si:
• 2 agents recommandent des actions différentes
• Différence de confiance > 0.7 entre agents
• Un agent identifie un risque ignoré par un autre
• Montant ≥ seuil vertical sans validation humaine

⚡ Modes de gel

GEL COMPLET → Tous les agents bloqués. Conflit critique, règle "block".

GEL DÉGRADÉ → Seul l'Agent Action est bloqué. Data et Raisonnement continuent d'enrichir le dossier d'arbitrage. L'humain reçoit un dossier plus complet.

CONSOLIDATION → Après un dégel, fenêtre de 5s pour capturer les résultats "en vol".

📐 Moteur de règles JsonLogic

Les règles sont des fichiers JSON purs. Pas de LLM. Pas d'incertitude.

// Exemple: comptable-001 — Aucune décision fiscale automatique
{

05 — Trust Box

🔐 Le concept produit

Inspiré d'Ant Group T-Box — La couche de confiance déterministe

🌊 Le Serment du Trust Box

« Ne jamais laisser l'IA prendre une décision seule dans les moments critiques. »

1. Déterminisme critique

Les décisions de gel et blocage sont 100% déterministes. Moteur JsonLogic pur. Pas de probabilité, pas d'incertitude. Si la condition est vraie, l'action est gelée. Point.

2. Gel préventif

Si une règle est violée, l'action est gelée AVANT exécution. L'Agent Action vérifie le statut de l'intention avant chaque traitement. C'est un fail-safe, pas un correctif.

3. Arbitrage humain obligatoire

L'IA ne décide JAMAIS seule pour les actions critiques. Le Médiateur gèle → l'humain décide. Chaîne d'escalade: expert → suppléant → associé. Timeouts: 2h → 4h → 8h.

4. Transparence totale

Chaque décision du Trust Box est tracée dans le journal WORM. On peut vérifier à tout moment l'intégrité de la chaîne de hash. L'endpoint /journal/verify le prouve.

5. Mode dégradé

En cas de gel dégradé, Data et Raisonnement continuent de travailler. Ils enrichissent le dossier d'arbitrage. L'arbitre humain a PLUS d'informations pour décider.

6. Conformité by design

RGPD, AI Act, secret professionnel FR-CH sont encodés dans les règles JsonLogic. Ce n'est pas une config optionnelle — c'est la base du système.

📜 Le Serment par profession

▶ ⚖️ Avocat — Secret professionnel absolu 
Je m'engage, en tant qu'agent numérique d'assistance juridique :

1. À préserver le SECRET PROFESSIONNEL ABSOLU.
   Art. 321 CP (Suisse), Art. 226-13 CP (France).

2. À fonctionner EXCLUSIVEMENT sur infrastructure locale certifiée.
   Aucune donnée client ne transite par un cloud public.

3. À ne JAMAIS rédiger de conclusions sans validation de l'avocat.

4. À respecter le principe du contradictoire.

5. À refuser toute instruction contraire aux règles déontologiques.

Références: Art. 321 CP, Art. 226-13 CP, Art. 47 LB, Loi 71-1130 Art. 66-5
▶ 🏥 Santé — Données patient protégées

En mode Haute Protection: LLM local Ollama, zero appel externe, données patient jamais hébergées en cloud. Conformité LPM + HDS. Gel automatique pour toute donnée sensible sans validation humaine.

▶ 🏦 Banque — Secret bancaire CH

Art. 47 LB: secret bancaire suisse. KYC renforcé ≥ 15K CHF. Conformité FINMA. Gel automatique pour transactions suspectes. Arbitrage humain obligatoire au-dessus du seuil.

06 — Sécurité

🔒 7 couches de sécurité

Chaque couche est indépendante et défend en profondeur

▶ 1 Authentification JWT + RBAC

Comment ça marche : Double token (access 30min + refresh 7j). Password hashé bcrypt. Refresh token stocké hashé en DB (jamais en clair). 4 rôles RBAC :

RôleUtilisateursIntentionsArbitrageAdmin
adminCRUDCRUD + freezeDécider + déléguer✅ Total
expertReadCRUDDécider❌
operator—Create + ReadRead❌
viewer—ReadRead❌

Clés API : Format cl_xxxx.... Hash SHA-256 stocké en DB. Préfixe visible pour identification. Expiration configurable. Scopes par clé.

Rate limiting : Login: 5 tentatives/minute par IP. Compteur réinitialisé après succès.

▶ 2 Journal Append-Only (WORM)

Comment ça marche : Chaque événement est écrit dans un fichier JSON-L (un par jour). Chaque entrée contient:

  • entry_hash = SHA-256(contenu + previous_hash)
  • signature = HMAC-SHA256(contenu + hash, signing_key)
  • previous_hash → chaîne vers l'entrée précédente
# Chaîne de confiance:
GENESIS → Entry#1 (hash_1) → Entry#2 (hash_2) → Entry#3 (hash_3) → ...
              ↑                   ↑                   ↑
         previous_hash=GENESIS  previous_hash=hash_1  previous_hash=hash_2

# Vérification d'intégrité: /api/v1/journal/verify
# Recalcule TOUTE la chaîne et compare chaque hash

Propriétés : Append-only (jamais de modification/suppression). Atomic write (fichier .tmp → rename). Persistance fichier (survit au redémarrage). Vérifiable à tout moment.

▶ 3 Circuit Breaker

Protège contre les agents défaillants. 3 états :

CLOSED (normal) → 5 échecs → OPEN (bloqué) → 60s → HALF_OPEN (test) →
    ↑                                                           |
    └──────────── 3 succès consécutifs ←──────────────────────┘
                                                    │
                                              1 échec → OPEN

Configurable par agent. Le CircuitBreakerRegistry maintient un breaker par agent.

▶ 4 Chiffrement Fernet (AES)

Chiffrement au repos des données sensibles (PII, tokens). AES-128-CBC + HMAC-SHA256. Clé dérivée du SECRET_KEY via SHA-256. Supporte la rotation de clé (max 2 clés actives).

Les champs PII (phone, mfa_secret) sont chiffrés avant stockage. Le flag {field}_encrypted = true indique qu'un champ est chiffré.

▶ 5 Verrous distribués Redis

L'Agent Action acquiert un lock Redis avant d'exécuter: intention:{id}. Auto-expiration 30s. Empêche l'exécution concurrente de la même intention. Le lock est TOUJOURS relâché dans un finally (même en cas d'erreur).

▶ 6 Guardrails (IN + OUT)

Le LLM Service applique des guardrails en 2 passes :

IN (input): Détection PII, injection de prompt, contenu non autorisé. Si bloqué → retourne erreur.

OUT (output): Vérification safety, conformité, pas de données sensibles dans la réponse. Si bloqué → réponse filtrée.

▶ 7 State Machine des intentions 
created → routed → processing → frozen → arbitrating → unfrozen → processing → completed
                                       ↓               ↑
                                  degraded_frozen → arbitrating → unfrozen
                                       ↓
                                   failed / cancelled

Chaque transition est validée. Impossible de passer de completed à frozen par exemple. L'historique complet de chaque intention est conservé (from, to, trigger, reason, timestamp).

11 — Quiz Audit

🧪 Teste ta compréhension

Les questions qu'un auditeur pourrait te poser — et les bonnes réponses

Q1: « Pourquoi le Médiateur n'utilise-t-il jamais de LLM ? »
A) Parce que les LLM sont trop lents pour le temps réel
B) Parce qu'une décision de gel doit être 100% déterministe et reproductible. Un LLM peut produire des résultats différents pour la même entrée, ce qui est inacceptable pour la conformité réglementaire.
C) Parce qu'on n'a pas les budgets pour les appels LLM
✅ Bonne réponse ! Le JsonLogic est déterministe: même entrée = même résultat, toujours. C'est essentiel pour la traçabilité réglementaire (RGPD Art. 22, AI Act). Un LLM pourrait ne pas geler une action critique à cause d'une variation aléatoire.
Q2: « Que se passe-t-il si l'Agent Data et l'Agent Raisonnement sont en conflit ? »
A) L'Agent Raisonnement a toujours raison car il utilise le LLM
B) Le Médiateur détecte le conflit (divergence de recommandation ou de confiance), gèle l'intention, et prépare un dossier contradictoire pour l'arbitrage humain.
C) Le système crash et redémarre
✅ Bonne réponse ! Le Médiateur compare les positions (recommendation, confidence, risks). Si divergence → gel (complet ou dégradé) + arbitrage. L'humain voit les positions contradictoires et décide.
Q3: « Comment prouvez-vous que le journal d'audit n'a pas été modifié ? »
A) On a des sauvegardes quotidiennes
B) Seul l'admin peut le modifier
C) Le journal est hash-chainé SHA-256. Chaque entrée contient le hash de la précédente. L'endpoint /journal/verify recalcule TOUTE la chaîne et compare. Si un seul bit a changé, la vérification échoue. De plus, chaque entrée est signée HMAC-SHA256.
✅ Bonne réponse ! Le journal est immuable par design (append-only = WORM). La chaîne de hash SHA-256 garantit l'intégrité. La signature HMAC garantit l'authenticité. L'atomic write (fichier .tmp → rename) garantit la résistance aux crashes.
Q4: « C'est quoi le "gel dégradé" et pourquoi il existe ? »
A) C'est quand le système fonctionne mal mais accepte quand même
B) C'est un mode où seul l'Agent Action est bloqué, mais Data et Raisonnement continuent d'enrichir le dossier d'arbitrage. L'arbitre humain reçoit un dossier PLUS complet pour prendre sa décision.
C) C'est un bug qu'on est en train de corriger
✅ Bonne réponse ! Le gel dégradé est un choix d'architecture délibéré. Au lieu de tout geler (l'humain attend avec un dossier incomplet), on laisse les agents de recherche travailler. Le résultat: un arbitrage plus informé, plus rapide, plus juste.
Q5: « Qu'est-ce que le Reflection Node et pourquoi est-il inspiré de JP Morgan ? »
A) C'est un deuxième appel LLM qui critique l'analyse du premier. JP Morgan a publié un papier ("Ask David") montrant qu'un LLM qui révise sa propre analyse améliore significativement la qualité. On applique ce pattern.
B) C'est un système de backup de la base de données
C) C'est le nom du Médiateur dans le code
✅ Bonne réponse ! Le Reflection Node est un pattern "LLM-as-judge". Après l'analyse du ReasoningAgent, un DEUXIÈME appel LLM (température basse = 0.3, rôle "réviseur") critique l'analyse. Si le delta de confiance > 0.3, l'analyse est révisée. C'est documenté, journalisé, et désactivable.
Q6: « Comment fonctionne le pattern Saga de l'Agent Action ? »
A) Chaque action est exécutée en parallèle pour plus de rapidité
B) Chaque action est une étape de saga enregistrée avec sa compensation. Les étapes s'exécutent séquentiellement. Si l'étape 3 échoue, on compense les étapes 2 puis 1 en ordre inverse. Toute la séquence est journalisée.
C) C'est un système de cache qui mémorise les actions réussies
✅ Bonne réponse ! Le Saga pattern garantit la cohérence transactionnelle. Exemple: envoyer email + notification + paiement. Si le paiement échoue → on rappelle l'email et on annule la notification. Chaque étape a sa fonction de compensation enregistrée AVANT exécution.
Q7: « Pourquoi avoir choisi LiteLLM comme couche d'abstraction LLM ? »
A) LiteLLM est model-agnostic: il supporte 18+ providers avec la même interface. On peut changer de modèle sans toucher au code. Le routing par verticale est configurable via .env. En mode Haute Protection, on bascule sur Ollama local sans modification.
B) C'est le seul provider gratuit
C) On a prévu de le remplacer bientôt
✅ Bonne réponse ! LiteLLM offre une abstraction provider-agnostic. On peut switcher entre OpenRouter, Anthropic, OpenAI, Ollama, DeepSeek, etc. sans changer une ligne de code métier. Le ModelRouter résout le modèle en fonction de la verticale et du mode de déploiement.

12 — Lexique

📖 Glossaire technique

Les termes que tu dois connaître par cœur

🏗️ NATS JetStream

Bus de messages persistant. Les agents communiquent par événements (publish/subscribe). JetStream ajoute la persistance et le replay.

📝 Append-Only / WORM

Write Once Read Many. Journal immuable: on ne fait qu'ajouter, jamais modifier ou supprimer. Format JSON-L.

🔗 Hash-chainé SHA-256

Chaque entrée du journal contient le hash de la précédente. Comme une blockchain simplifiée. Si on modifie une entrée, toute la chaîne se brise.

🔑 JWT + RBAC

JSON Web Token + Role-Based Access Control. 4 rôles (admin, expert, operator, viewer). Access token 30min, refresh 7 jours.

🔐 Fernet (AES)

Chiffrement symétrique AES-128-CBC + HMAC-SHA256. Utilisé pour chiffrer les données sensibles au repos (PII, secrets).

⚡ Saga Pattern

Pattern transactionnel: chaque étape enregistre sa compensation. Si échec → compensation en ordre inverse. Garantit la cohérence.

🔌 Circuit Breaker

Protection: si un agent échoue 5 fois → OPEN (bloqué) pendant 60s → HALF_OPEN (test) → recovery ou re-blocage.

📐 JsonLogic

Moteur de règles en JSON pur. Déterministe: même entrée = même résultat. Utilisé par le Médiateur. 0% LLM.

🧬 RAG (ChromaDB)

Retrieval-Augmented Generation. Recherche sémantique dans les embeddings vectoriels. Enrichit le contexte LLM avec des faits réglementaires.

🤖 LiteLLM

Abstraction provider-agnostic. Même code pour OpenRouter, Anthropic, Ollama, OpenAI, etc. 18+ providers supportés.

🛡️ Guardrails

Filtres IN (PII, injection) et OUT (safety, compliance) autour des appels LLM. Bloque les contenus non conformes.

🔍 Reflection Node

Auto-critique LLM (pattern JP Morgan "Ask David"). Un 2e appel LLM révise l'analyse du 1er. Ajuste le score de confiance si nécessaire. v5.3: injecte les échecs passés et leçons apprises.

🧠 ProceduralMemory

v5.3 — Instructions apprises par agent × verticale. Un agent qui a traité 50 dossiers "avocat" accumule des leçons qui enrichissent automatiquement ses prompts futurs.

🔴 FailureMemory

v5.3 — Journal des échecs agents. Chaque erreur est enregistrée avec cause racine + correction. Les erreurs répétées (3x) déclenchent des alertes avant les missions futures.

🏗️ Context Builder

v5.3 — Constructeur de contexte en 5 couches: Système → Vertical → Procédural → Mémoire → Tâche. Combine toutes les sources pour composer un prompt enrichi par agent.

📊 Agent Tracer

v5.3 — Tracing agent-level JSONL. Chaque action agent est tracée avec timing, tokens, coût. Métriques P50/P95, taux de succès par agent/verticale. Complémentaire OTel + Prometheus.
